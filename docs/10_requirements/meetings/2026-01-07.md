### 3.3 EventStorming 진행 방식

#### (1) 요구사항 도출
- 2026.01.07 회의에서 진행한 정리에서 작성된 **명세서가 있으므로**, 이를 **베이스로 활용**한다.

#### (2) Domain Event 도출 기준
- **User Flow**: 사용자의 클릭/입력에 의해 발생하는 이벤트
- **System Process**: 사용자가 모르는 뒷단에서 발생하는 이벤트
- **Exception**: 도메인에서 발생할 수 있는 큰 예외 이벤트

#### (3) Command 및 Actor 식별

**Command**
- Command는 보통 **‘사용자의 의도’**를 가진다.
- 추후 구현할 **API 엔드포인트와 1:1로 매핑**되는 경우가 많아,  
  API 설계를 미리 진행해본다는 느낌으로 접근한다.
- DDD 원칙상 Command는 동사형이 일반적이지만,  
  이번 설계에서는 **도출 편의상 명사형으로 우선 정의**하고,  
  추후 구현 단계에서 **메서드명으로 구체화**한다.
- 반드시 1개의 Command가 1개의 Event를 생성하지는 않는다.

**Actor**
- Command를 실행하는 주체(Actor)를 식별한다.
- Actor는 반드시 사람일 필요는 없다.  
  시스템 내부 트리거나 AI 모델 등도 Actor로 식별 가능하다.
